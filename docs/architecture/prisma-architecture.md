# Prisma ORM — Architecture & Workflow

This document is the single map of every Prisma-related file in this service: where it lives, what owns it, and the exact commands to run against it. Read this before touching the schema, migrations, or the generated client.

> See also: [Commands](../commands/prisma.md) for the exact CLI commands, [Concepts](../concepts/prisma.md) for how repositories, transactions, pagination, and enums are actually used in app code.

## 2. How the pieces connect

```
.env* files
   │  (DATABASE_URL)
   ▼
prisma.config.ts ──loads env, in priority order──▶ used by Prisma CLI only
   │                                                 (migrate, studio, generate, validate, db ...)
   ▼
prisma/schema/*.prisma ──merged, then `prisma generate`──▶ src/generated/prisma/* (PrismaClient + types)
                                                 │
                                                 ▼
                                   src/prisma/prisma.service.ts
                                   - reads ConfigService.get('database.url')
                                   - builds a `pg.Pool` + `PrismaPg` driver adapter
                                   - extends the generated PrismaClient with that adapter
                                                 │
                                                 ▼
                                   src/prisma/prisma.module.ts (@Global)
                                   - ConfigModule.forFeature(databaseConfig)
                                   - provides/exports PrismaService app-wide
                                                 │
                                                 ▼
                          Feature repositories (extends BaseRepository)
                          - inject PrismaService
                          - call this.prisma.<model>.findMany(...) etc.
                          - use this.withTransaction(fn) for multi-step writes
```

Key point: **the Prisma CLI (`prisma migrate`, `prisma studio`, `prisma generate`) and the running NestJS app read `DATABASE_URL` through two completely separate paths.**

- CLI commands → `prisma.config.ts` → its own `dotenv` loading.
- The running app → `src/app.module.ts`'s `ConfigModule.forRoot({ envFilePath: [...] })` → `src/prisma/config/database.config.ts` → `PrismaService` via `ConfigService`.

Both now load the **same files in the same priority order** (see [Env file precedence](../concepts/prisma.md#env-file-precedence)), but they are still two independent code paths — if you ever change one, change the other to match.

## 3. Multi-file schema structure

The schema is split by domain under `prisma/schema/`, not a single `schema.prisma` file. Prisma 7's multi-file support merges every `*.prisma` file in the configured folder (`prisma.config.ts` → `schema: 'prisma/schema'`) as if it were one file — cross-file relations (e.g. `Product` → `Category`) resolve normally, no imports needed.

```prisma
// prisma/schema/schema.prisma — generator + datasource ONLY, nothing else
generator client {
  provider     = "prisma-client"        // new (v7) client generator, not the legacy "prisma-client-js"
  output       = "../../src/generated/prisma"  // one level deeper than a single-file schema would need
  moduleFormat = "cjs"
}

datasource db {
  provider = "postgresql"
  // NOTE: no `url` here — the connection string is supplied at runtime by
  // prisma.config.ts (CLI) and PrismaService (app), not by this file.
}
```

> ⚠️ **Relative path gotcha:** because `schema.prisma` now lives at `prisma/schema/schema.prisma` instead of `prisma/schema.prisma`, the generator `output` path needed an extra `../` (`../../src/generated/prisma`) to still land at `src/generated/prisma`. If you ever move this file, re-run `npx prisma generate` and check the output path printed in the CLI log.

**Where each domain lives:**

| File | Models | Enums |
|---|---|---|
| `user.prisma` | `User`, `Profile`, `UserSecurity`, `Session`, `OTP` | `UserRole`, `AuthProvider`, `UserStatus`, `OTPType` |
| `category.prisma` | `Category` | — |
| `product.prisma` | `Product`, `ProductVariant`, `ProductImage` | `ProductType`, `DiscountType`, `StockStatus` |
| `inventory.prisma` | `Inventory` | `InventoryExchangeType` |
| `shared.prisma` | — | `CategoryProductStatus` (used by **both** `Category.status` and `Product.status` — that's why it's not in either domain file) |

**Rule for new domains:** add a new `<domain>.prisma` file named after its `src/modules/<domain>` counterpart. If a new enum or model is genuinely used by 2+ domain files, it goes in `shared.prisma` — never duplicate it into each file that needs it.

Each model uses `@@map`/`@map` to keep Prisma's camelCase field names mapped to `snake_case` Postgres columns/tables.

## 4. Conventions to follow

- **Never** edit anything under `src/generated/prisma/`. It is fully regenerated by `prisma generate` / `migrate dev` and any manual edits will be silently lost.
- **Never** hand-edit a folder under `prisma/migrations/`. If a migration was applied wrong, write a new forward migration — don't mutate history that may already be applied elsewhere.
- All new repositories should extend `src/prisma/base.repository.ts` (`BaseRepository`) and inject `PrismaService`, so multi-step writes can use `this.withTransaction(fn)` instead of ad-hoc `$transaction` calls.
- `DATABASE_URL` is only ever read in two places: `prisma.config.ts` (CLI) and `src/prisma/config/database.config.ts` (app, via `PrismaService`). Don't read `process.env.DATABASE_URL` anywhere else.
